# Minimal Conflict Certificates

Stage 6 handles negotiations where no common hard-safe profile exists. The
protocol fails closed: it does not select a profile, create a trust contract,
activate TLS, resume a session, or attempt fallback.

The implementation distinguishes three terms precisely:

- A Z3 unsat core is a solver-provided subset of tracked assertions explaining
  unsatisfiability.
- An irreducible unsatisfiable subset, or IUS, is unsatisfiable and becomes
  satisfiable when any individual member is removed.
- A minimum-cardinality unsatisfiable subset is globally smallest. Stage 6 does
  not claim to find this.

The certificate uses deterministic deletion-based shrinking over the sorted Z3
core. The result is a subset-minimal conflict certificate, not a global
minimum-cardinality proof.

Only normalized conflict-relevant constraints are serialized. Private policy
bodies, private keys, and nonces are not written to conflict artifacts.

Remediation reports are informational. They test possible removals or
weakening candidates independently, never mutate policy or manifests, and never
create contracts.

The complete Stage 6 artifact bundle is generated by
`scripts/validate_stage6.py`. It owns the atomic replacement of
`artifacts/conflicts/`, combines conflict certificates, safe-abort records,
failure transcripts, remediation reports, validation reports, and
`checksums.sha256`, and verifies the checksum inventory before publishing the
replacement. The lower-level component scripts write to separate component
directories by default and do not replace the bundle.

Checksum validation is an integrity check, not the scientific validation
criterion. The bundle also validates report type discriminators, expected
scenario IDs, artifact counts, IUS-derived taxonomy, and cross-artifact
references before atomic replacement. A bundle with correct hashes but reused
or incomplete report payloads is invalid.

Each infeasible scenario must produce the complete chain:

1. Minimal conflict certificate.
2. Failure transcript referencing the certificate hash.
3. Safe-abort record referencing the same session and transcript hash.
4. Informational remediation report.

The registered infeasible scenarios are `no-common-profile`,
`assurance-floor-conflict`, `TLS-group-capability-conflict`,
`lease-policy-conflict`, and `multi-cause-conflict`. The assurance-floor
scenario keeps the pre-policy capability intersection non-empty and derives
`ASSURANCE_FLOOR_CONFLICT` from an assurance-only IUS. The lease scenario keeps
capability, assurance, TLS group, endpoint authentication, evidence algorithm,
fallback, and resumption otherwise satisfiable, then derives `LEASE_CONFLICT`
from lease-bound constraints only.

Feasible regression is intentionally separate from safe-abort validation. It
checks the four Stage 5 feasible contracts, verifies the selected profile and
signed-contract hash are unchanged, and confirms no certificate, failure
transcript, or abort record is produced.

Adversarial validation is also separate. It covers the registered mutation
matrix: removed conflict constraint, added unrelated constraint, modified
source hash, false empty-common-safe-set claim, false IUS minimality claim,
satisfiable set presented as unsatisfiable, modified conflict category,
modified certificate hash, modified failure transcript, selected profile after
abort, trust contract after abort, fallback changed to true, replayed failure
session, certificate from another session, and tampered commit-reveal
transcript.
# Stage 7 Infeasible Runtime Binding

In Stage 7, verified infeasible reveals proceed to the existing deterministic
conflict certificate, failure transcript, and safe-abort record. The execution
gate rejects the session, the TLS executor is not invoked, and no task request is
accepted.
